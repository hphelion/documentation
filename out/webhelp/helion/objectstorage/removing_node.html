HPE Helion OpenStackÂ® 2.0: Removing a Swift Node

You can use this process when you want to remove one or more nodes permanently from the Swift system. This process has three phases:

  • Removing the node's drives from the rings - move data off the node and store it on the remaining nodes.
  • Disabling Swift on the node - prevent the node from participating in the Swift system.
  • Removing the node from the input model - the node is removed completely from the system.

To Remove a Node's Drive from the Rings

To remove a node's drives from the rings, change the role of the server that is being removed so that it uses that disk model that does not include Swift drives. Then a new ring is built and deployed. The new rings will not contain the node's drives.

  1. Edit the definition of the server (usually in the data/servers.yml file). For example, if the entry for the server looks like:
    - name: SWOBJ-DISKS
- id: swobj4
  ip-addr: 192.168.10.5
  role: SWOBJ-ROLE
  server-group: RACK3  
   ...
    Change the role to a new value, for example, SWOBJ-DRAINING-ROLE:
    - name: SWOBJ-DISKS
- id: swobj4
  ip-addr: 192.168.10.5
  role: SWOBJ-DRAINING-ROLE
  server-group: RACK3
   ...
  2. Edit the server-roles definition (usually in the data/server_roles.yml file) and add the SWOBJ-DRAINING-ROLE role. Keep the interface model of the original SWOBJ-ROLE, but change the disk model to SWOBJ-DRAINING-DISKS:
    - name: SWOBJ-ROLE
  interface-model: INTERFACE_SET_SWOBJ
  disk-model: SWOBJ-DISKS
 
- name: SWOBJ-DRAINING-ROLE
  interface-model: INTERFACE_SET_SWOBJ
  disk-model: SWOBJ-DRAINING-DISKS
  3. Create a SWOBJ-DRAINING-DISKS disk model by editing the file containing the SWOBJ-DISKS disk model (usually the data/swobj_disks.yml file) and adding an entry for SWOBJ-DRAINING-DISKS. Make a copy of the SWOBJ-DISKS disk model, but change the consumer field from swift to swift-draining. For example:
    disk-models:
- name: SWOBJ-DISKS  
  volume-groups:
    - name: hlm-vg
      ...
      consumer:
         name: os
  device-groups:
    - name: swiftobj
      devices:
        - name: /dev/sdb
        - name: /dev/sdc
        - name: /dev/sdd
      consumer:
        name: swift
        attrs:
          rings:
            - object-0
 
- name: SWOBJ-DRAINING-DISKS  
  volume-groups:
    - name: hlm-vg
      ...
      consumer:
         name: os
  device-groups:
    - name: swiftobj
      devices:
        - name: /dev/sdb
        - name: /dev/sdc
        - name: /dev/sdd
      consumer:
        name: swift-draining
        attrs:
          rings:
            - object-0
    Note: Before you run any playbooks, remember that you need to export the encryption key in the following environment variable: export HOS_USER_PASSWORD_ENCRYPT_KEY=<encryption key>. For more information, see HPE Helion OpenStack 2.0: Installation for HPE Helion Entry-scale Cloud with Swift Only.
  4. Commit the changes to Git:
    cd ~/helion
git commit -a -m "<commit message>"
  5. Run the configuration processor:
    cd ~/helion/hos/ansible
ansible-playbook -i hosts/localhost config-processor-run.yml
  6. Create a deployment directory:
    cd ~/helion/hos/ansible
ansible-playbook -i hosts/localhost ready-deployment.yml
  7. Run the swift-deploy.yml playbook. Do not limit this process to just the node (helion-cp1-swobj4) that you are draining:
    cd ~/scratch/ansible/next/hos/ansible
ansible-playbook -i hosts/verb_hosts swift-deploy.yml
  8. Wait until replication has finished. For more information, see HPE Helion OpenStack 2.0: Determining When to Rebalance and Deploy a New Ring.

    At this stage, all data has been moved from helion-cp1-swobj4 and replicated on the remaining nodes. Now, you can proceed to the next phase.

To Disable Swift on a Node

The node is still running Swift services. It is still part of the input model and is fully configured so that you can log in and perform the other operations.

To disable the node, use the following steps. In this example, swobj4 is the node being removed from Swift.
  1. Stop Swift services on the node using the swift-stop.yml playbook:
    cd ~/scratch/ansible/next/hos/ansible
ansible-playbook -i hosts/verb_hosts swift-stop.yml --limit <hostname>
    The following example uses the swift-stop.yml playbook to stop Swift services on helion-cp1-swobj0004:
    cd ~/scratch/ansible/next/hos/ansible
ansible-playbook -i hosts/verb_hosts swift-stop.yml --limit helion-cp1-swobj0004
    Note: When using the --limit argument, you must specify the full hostname (for example: helion-cp1-swobj0004) or use the wild card * (for example: *swobj4*).
  2. Remove the configuration files. This process assumes you are logged onto the Deployer:
    ssh helion-cp1-swobj4-mgmt sudo rm -R /etc/swift
    Note: Do not run any other playbooks until you have finished the process described in To Remove a Node from the Input Model. Otherwise, these playbooks may recreate /etc/swift and restart Swift on swobj4. If you accidentally run a playbook, simply repeat the process in To Disable Swift on a Node.

To Remove a Node from the Input Model

Use the following steps to remove a node.
  1. Edit the servers list (usually in data/servers.yml file). Remove the entry for the node (swobj4 in this example).
  2. If this was a SWPAC node, reduce the member-count attribute by 1 in the control-planes definition (usually in the data/control_plane.yml file). For SWOBJ nodes, no such action is needed.
  3. Commit the changes to Git:
    cd ~helion
git commit -a -m "<commit message>"
  4. Run the configuration processor:
    cd ~/helion/hos/ansible
ansible-playbook -i hosts/localhost config-processor-run.yml
  5. Create a deployment directory:
    cd ~/helion/hos/ansible
ansible-playbook -i hosts/localhost ready-deployment.yml
  6. Remove the node from Cobbler:
    sudo cobbler system remove --name=swobj4
ansible-playbook -i hosts/localhost cobbler-deploy.yml
  7. If the node was a SWPAC node, run the hlm-deploy.yml playbook:
    cd ~/scratch/ansible/next/hos/ansible
ansible-playbook -i hosts/verb_hosts hlm-deploy.yml
  8. Wait until replication has finished. For more details, see HPE Helion OpenStack 2.0: Determining When to Rebalance and Deploy a New Ring.
  9. You may need to continue to rebalance the rings. For instructions, see Final Rebalance Phase at HPE Helion OpenStack 2.0: Applying Input Model Changes to Existing Rings.
