HPE Helion OpenStackÂ® 2.0: Creating the HPE Helion Development Platform Service Net

Many of the HPE Helion Developer Platform services require the creation of a VM service network in order to access the cloud control plane. This service network is a VLAN-encapsulated provider network which is fully routable to the HPE Helion OpenStack MGMT network.
Important: Create this network before attempting to install the HPE Helion Developer Platform.
  1. SSH to the deployer/lifecycle-manager node using admin credentials.
  2. Source the admin credentials: 
    source ~/service.osrc
  3. Configure a Neutron network that will map to an external network for the service net.
    • For simple configurations, a lifecycle-manager playbook has been provided for this purpose.
    • For non-trivial network configurations, any additional routes should be added to the subnet definition using the --host-route option of the neutron subnet-create command. The --host-route option can be specified multiple times if necessary. 
      cd ~/scratch/ansible/next/hos/ansible
ansible-playbook -i hosts/verb_hosts neutron-svc-net-configure.yml \
    -e PROV_NET_TYPE=<provider_network_type> \
    -e PROV_PHYS_NET=<physical_network_name> \
    -e PROV_NET_VLAN_ID=<vlan_id> \
    -e SVC_NET_CIDR=<provider_network_cidr> \
    -e SVC_NET_DEST=<svc_net_destination> \
    -e SVC_NET_GATEWAY="gateway_used_by_the_network"
      Example: Given a management network with the following attributes:
      • MGMT Net = 192.168.245.1/32
      • SVC Net = 10.55.66.0/24
      • SVC gateway to MGMT = 10.55.66.3
      • SVC Net VLAN = 100
      the playbook command for creating the service network would be: 
      cd ~/scratch/ansible/next/hos/ansible
ansible-playbook -i hosts/verb_hosts neutron-svc-net-configure.yml -e PROV_NET_TYPE=vlan -e PROV_PHYS_NET=physnet1 -e PROV_NET_VLAN_ID=100 -e SVC_NET_CIDR=10.55.66.0/24 -e SVC_NET_DEST=192.168.245.1/32 -e SVC_NET_GATEWAY=10.55.66.3
    Note: To configure a service network for flat encapsulation, modify the service network creation statement: -e PROV_NET_TYPE flat The subnet creation statement does not change.
  4. Configure your my_cloud definition to include the service network. This is used to provide routing from management (and potentially other networks) back to the newly created service net.
    1. Ensure that the service network is routable from the management network and that the service network is defined in ~/helion/my_cloud/definition/data/network_groups.yml: 
      # Management 
# 
# This is the network group that will be used for 
# management traffic within the cloud. 
# 
# The interface used by this group will be presented 
# to Neutron as physnet1, and used by provider VLANS 
# 
# In this example this group is also used for ISCSI 
# traffic between VMs and the VSA block storage 
# 
- name: MANAGEMENT 
  # removed items 
  routes:
    - SERVICE 
  # removed items 
# Service 
# 
# This is the network group that will be used for in-cloud 
# service traffic to control plane 
# 
- name: SERVICE
    2. Ensure that there is a valid definition for the service network in ~/helion/my_cloud/definition/data/networks.yml
      Note: The cidr and gateway given must match the values provided when the network was created.
      - name: SERVICE-NET
  vlanid: 306 
tagged-vlan: true 
  cidr: 10.1.6.0/24 
  gateway-ip: 10.1.6.1 
  network-group: SERVICE
    3. Ensure that this network is a member of the server-group listed in ~/helion/my_cloud/definition/data/server_groups.yml: 
      server-groups: 
- name: CLOUD 
  server-groups: 
    - AZ1
    - AZ2
    - AZ3
  networks: 
    - EXTERNAL-NET
    - GUEST-NET 
    - MANAGEMENT-NET
    - PXE-NET 
    - SERVICE-NET 
# 
# Create a group for each availability zone 
#
- name: AZ1 
  server-groups: 
    - RACK1 
- name: AZ2 
  server-groups: 
    - RACK2 
- name: AZ3
  server-groups:
    4. Once these changes have been made, the normal mechanisms should be used (configuration-processor-run.yml, ready-deployment.yml, site.yml) to push this change to production.
