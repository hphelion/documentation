HPE Helion OpenStack® 2.0: Replacing a Controller Node

For HPE Helion OpenStack 2.0 versions, you must have three controller nodes. Therefore, adding or removing nodes is not an option. However, if you need to repair or replace a controller node, you may do so by following the steps outlined here. Note that to run any playbooks whatsoever for cloud maintenance, you will always run from the deployer/lifecycle-manager node.

Note: Do not add entries for a new server; instead, just update the entries for the broken one. Also note that if you are replacing a control plane node that also happens to be running the deployer, that process is different. You’ll need to restore you deployer data first.
Note: Be aware that all management commands are run on the node where the deployer is running.
Update your model with replacement server information
  1. Update your cloud model (servers.yml ) Update your cloud model (servers.yml ) with new mac-addr, ilo-ip, ilo-password, ilo-user fields where these have changed. Do not change the id, ip-addr, role, server-group settings. (Please follow the procedure for updating your cloud model/ git repo)
  2. Get the servers.yml file stored in git: 
    cd ~/helion/my_cloud/definition/data
git checkout site
    then change, as necessary, the mac-addr, ilo-ip, ilo-password, and ilo-user fields of this existing controller node. Save and commit the change 
    git commit -a -m "repaired node X"
  3. Run the configuration processor as follows: 
    cd ~/helion/hos/ansible
ansible-playbook -i hosts/localhost config-processor-run.yml
    Then run ready-deployment: 
    ansible-playbook -i hosts/localhost ready-deployment.yml
Image the new node
  1. Log onto the node that is running the deployer. Remove the broken control plane node from cobbler by running sudo cobbler system remove --name <server-id>. Here's an example:
     sudo cobbler system remove --name controller2
  2. Remove the ssh key of the old controller node from the known hosts file. Here's an example: 
    ssh-keygen -f "/home/stack/.ssh/known_hosts" -R 10.123.12.3
  3. Run the cobbler deploy playbook to add the replacement node 
    ansible-playbook -i hosts/localhost cobbler-deploy.yml
  4. Re-image the new nodes by running bm-reimage.yml: ansible-playbook -i hosts/localhost bm-reimage.yml -e nodelist=<node-name>. Note: Note: you must ensure that the old controller node is powered off before completing this step. This is because the new controller node will re-use the original IP address. 
    ansible-playbook -i hosts/localhost bm-reimage.yml -e nodelist=controller2
  5. Configure the necessary keys used for the database etc: 
    cd ~/scratch/ansible/next/hos/ansible
ansible-playbook -i hosts/verb_hosts monasca-rebuild-pretasks.yml
  6. Log into the 2 original controller nodes as "dbadmin" and run the following command: 
    ssh-keygen -f "/home/dbadmin/.ssh/known_hosts" -R {{ip_of_node_to_replace}}
  7. Run osconfig on the replacement controller node. For example: 
    ansible-playbook -i hosts/verb_hosts osconfig-run.yml  -e rebuild=True --limit=hlm-cp1-c1-m2-mgmt
  8. If the controller being replaced is the first server running the swift-proxy service (see Identifying the First Proxy Server) you need to restore the Swift Ring Builder files to the /etc/swiftlm/builder_dir directory. See Recovering Builder Files for details.

  9. Run swift-deploy across all controllers using: 
    ansible-playbook -i hosts/verb_hosts swift-deploy.yml
  10. Run hlm-deploy on the replacement controller 
    ansible-playbook -i hosts/verb_hosts hlm-deploy.yml -e rebuild=True --limit=hlm-cp1-c1-m2-mgmt
